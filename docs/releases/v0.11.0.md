# Koalafrog HQ v0.11.0

Status: **In development**

## Release foundation

- Corrected new backup exports to record the application version supplied from `package.json` at build time.
- Preserved `koalafrog-backup-v1` and `koalafrog-hq:workspace:v9`; older application-version metadata remains valid.
- Restored discoverable access to Platform and backup tools through the subordinate **System** navigation group.
- Added focused coverage for version metadata, legacy backup validation, and Platform navigation.

## Beard Butter Product Studio

- Added Beard Butter as the first heat-processed, anhydrous, multi-phase Product Studio archetype.
- Added ordered Formula phase metadata for heat, cool-down, and temperature-sensitive additions without forcing existing single-phase formulas to change.
- Added structured draft manufacturing steps with temperature ranges, mixing guidance, critical-step markers, and completion criteria.
- Added practical 1–5 development comparison fields for texture, defects, application feel, conditioning, appearance, stability, scent, and overall result.
- Formula handoff creates the existing Product, Formula, editable Draft Version, and phase-aware Formula Lines. Lab creation snapshots planned weights and ordered process controls; planning creates no inventory movement.
- Added additive relational columns for optional Formula phase/process metadata and Lab execution observations. Existing records receive empty/default metadata and remain valid.

Known limitations: recommendations are structured development guidance, not validated manufacturing instructions. Supplier-specific melting behavior, usage limits, stability, microbiological risk, safety assessment, CPSR conclusions, and regulatory suitability require competent external review and physical evidence.

The Beard Butter slice requires additive migration `20260718120000`; it does not change the local workspace schema identifier or backup format.

The retained v9 JSON envelope remains backward-compatible for current code because every new field is optional and existing v9 backups hydrate without defaults overwriting records. New phase/process metadata is preserved by backup and repository round-trips. Application-version metadata is not a compatibility gate, so an older application build must not be used to edit Beard Butter records merely because it accepts the v9 envelope; use the current release for records containing these fields.
